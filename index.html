SECTION 2 / 95

C'est quoi SAGS OS ?

SAGS OS, c'est une application web que j'ai développée pour centraliser toute la gestion du Groupe SAGS. Au lieu d'avoir des fichiers Excel, des mails, des appels dans tous les sens — tout est regroupé dans une seule interface.

Concrètement, il y a deux interfaces : une pour l'admin (toi Jalil) et une pour tes agents terrain (sur leur téléphone). Plus un chatbot IA intégré pour les deux.

L'app tourne sur visionia-france.com, hébergée chez Vercel à Paris (datacenter CDG1). Base de données Supabase. Intelligence artificielle Claude (Anthropic) + GPT-4o (OpenAI).

SECTION 19 / 95

Sous le capot : l'IA de génération

Voici le pipeline V2 complet — ce qui se passe quand tu cliques « Générer » :

1
Extraction & diagnostics CCTP

unpdf (PDF page par page avec marqueur [PAGE N]) ou mammoth (DOCX). Détection des CCTP « écrasés » avec re-extraction. Diagnostics : caractères, mots, lignes, pages, blocs.

2
Profilage CCTP (cctpProfile)

Extraction structurée : sites nominatifs, horaires, livrables, délais, contraintes opérationnelles, termes de validation. Détection de la famille de marché.

3
Plan canonique + Blueprint métier

Construction du plan : mode libre/imposé, objectif pages/mots, matrice exigences → sections, schémas (organigrammes, plannings, logigrammes). Résolution du blueprint par famille (titres, sous-sections, lexique hors-sujet).

4
RAG Supabase (FTS + tags marché)

Recherche Full-Text via search_rag_chunks_fts avec biais de récence et boost qualité. Filtrage par tags marché. Le CCTP reste la source primaire, le RAG enrichit.

5
Rédaction par Claude Sonnet 4.6

Génération par lots (batching). Prompt système avec : CCTP = source primaire, sites/horaires/délais explicites, accents obligatoires, zéro placeholder, claims prouvés uniquement. Temp 0.35.

6
Double review : GPT-5.4-mini + Claude Sonnet 4.6

Deux reviewers jugent en parallèle. Issues et patchs fusionnés. Score le plus sévère retenu. Critères bloquants : mémoire générique, sites manquants, horaires absents, placeholders, accents, hors-sujet.

7
Boucle corrective (max 3 itérations)

Si score < 55/60 ou blocages : Claude Sonnet 4.6 corrige sur base des patch_instructions, puis re-review. Jusqu'à validation ou plafond.

8
Quality gates + export

Validation export : placeholders, accents métier, vocab hors-sujet, claims non prouvés, couverture CCTP (sites, horaires, livrables, termes). Si échec = export bloqué.

erp_musee

SECTION 25 / 95

La boucle de double review (review-loop.ts)

C'est le cœur du contrôle qualité. Deux modèles IA indépendants jugent le mémoire. La lecture la plus sévère gagne. Si échec, on corrige et on recommence.

Reviewer OpenAI

Primaire : gpt-5.4-mini

Fallback 1 : gpt-5-mini

Fallback 2 : gpt-4o

Reviewer Anthropic

Modèle : claude-sonnet-4-6

Même modèle que le rédacteur, mais avec un prompt reviewer spécifique.

Règle de fusion

•Les issues des deux reviewers sont fusionnées
•Les patchs sont fusionnés
•Le score le plus sévère est retenu
•Le verdict mentionne les deux scores
•Score cible : 55/60
•Max 3 itérations de correction
•Zéro blocage requis (REQUIRE_NO_BLOCKING=true)

Sites nominatifs absents

Horaires / délais manquants

Obligations temporelles du CCTP non reprises

Livrables critiques oubliés

Rapports, bilans, annexes demandés

Placeholders visibles

« À compléter », « Si retenu », « Cf annexe »

Accents manquants

« securite » au lieu de « sécurité »

SECTION 60 / 95

Intégration RAG

Chaque document GED est automatiquement indexé dans pgvector pour la recherche sémantique. Le contenu est découpé en chunks de 3200 caractères avec un overlap de 400, embed en vecteurs 1536 dimensions, et stocké avec un index HNSW. PARTIE 5

Architecture Technique

Supabase, pgvector, API routes, sécurité, déploiement. Le dessous des cartes.

SECTION 61 / 95

Les 3 clients Supabase

L'app utilise 3 instances différentes de Supabase selon le contexte :

Client (Browser)

Pour les requêtes côté navigateur. Utilise la clé anon. Cookies gérés automatiquement par @supabase/ssr.

Server (SSR)

Pour les Server Components et Server Actions. Cookie middleware avec try-catch pour la compatibilité read-only.

Admin (Service Role)

Pour les opérations sensibles : bypass RLS, gestion des utilisateurs, RAG ingestion. Stateless, pas de session.

SECTION 62 / 95

Le middleware d'authentification

Chaque requête passe par un middleware qui vérifie la session et redirige selon le rôle. Un agent terrain ne peut pas accéder au dashboard, et un admin ne peut pas accéder au portail agent.

SECTION 63 / 95

pgvector : la recherche sémantique

Le RAG (Retrieval-Augmented Generation) fonctionne avec pgvector, une extension PostgreSQL pour les vecteurs. Chaque document est découpé en morceaux, transformé en vecteur 1536 dimensions, et indexé avec HNSW (m=16, ef=64). La recherche utilise la distance cosinus avec un seuil de 0.7 et un boost de 1.2x pour les documents récents.

SECTION 64 / 95

Chunking : comment on découpe les textes

Stratégie de chunking en 3 niveaux : d'abord par headers markdown (##), puis par paragraphes, puis par phrases. Taille cible : 3200 caractères (~800 tokens) avec 400 caractères d'overlap entre les chunks pour garder le contexte.

Claude (Anthropic)

SECTION 69 / 95

Server Actions : le pattern

Les 22 fichiers de server actions suivent tous le même pattern :

"use server"

1. Auth check → supabase.auth.getUser()
2. Zod validation → schema.safeParse(data)
3. Supabase query → createClient() ou createAdminClient()
4. Error return → {error: "message"} ou {success: true}
5. Revalidation → revalidatePath("/dashboard/xxx")

SECTION 70 / 95

Notifications temps réel

Les notifications utilisent Supabase Realtime. Quand une notification est insérée dans la table, le hook useNotifications met à jour le compteur de la cloche en temps réel via WebSocket. Pas de polling, pas de refresh.
